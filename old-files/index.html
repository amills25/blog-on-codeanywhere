Week Seven

Technical Article

Computer programming is a lot more than just writing code. If you jump directly into code when presented with a new project, you're likely to end up with numerous bugs and create more work for yourself. It is best practice, instead, to write pseudocode before actual code. Pseudocode is merely writing out the patterns, steps, and algorithms of a program in plain English.

By doing this, you can logically think through your program and decide how you want to make it work, which can then be translated into any coding language you want. I like to incorporate a little bit of real code in mine and start translating pieces as I think them through because I know I'll have to do so eventually, but for many, keeping it in plain English is most beneficial. In those cases, you could hand your pseudocode to someone who doesn't know anything about programming, and they would be able to read and understand what your program is going to do

Having been recently introduced to Object Oriented Programming, I found that my pseudocode really helped me more than it had in the past. When you are working with objects and classes in JavaScript, it is vitally important to properly organize your thoughts and your code methods. Otherwise, you will get lost, pieces won't connect, and your program will be full of bugs.

What I didn't expect, though, was that Object Oriented Programming helped me with my pseudocode. Knowing that my program needed to be properly organized to function correctly, I was better able to write pseudocode knowing I had to decide in advance where pieces of code would be and how they would connect.

This method of organization is better known as a design pattern.

This week in web developer bootcamp, we were introduced to design patterns for the first time. While we used a couple of them, including Model-View-Controller, my preferred design pattern at this time is the Composite pattern. In the composite pattern, you "compose" objects into "parent" and "child" structures so you can treat each of them as one. Much like a human child inherits attributes from their parents, this coding design pattern acts in a similar fashion. Your parent component can make requests from its child and it can act accordingly because it has inherited the same traits as the parent.

Another way to look at this pattern is with a tree and leaves. The main object component of your program is the tree and a leaf is another object that inherits the traits of the tree and can do actions at the request of the tree. As a parent with multiple child objects, or a tree with multiple leaf objects, it would make sense to have a method in your main component that would run on each of the children at the same time to create the same effect on each of them.

Naturally, with having dozens of design patterns available, there are good and bad times to use this particular one. You would choose the composite pattern if you are using multiple objects in the same way, and know you're going to have very similar code to work on each of them. Each child is going to inherit attributes from the parent, therefore you are able to work similarly on each of them. I found it beneficial this week when creating a Tic-Tac-Toe game to use this pattern because I created a game object as a parent, and a tile object as a child. If I had decided to use 4 different objects that each work in different ways, then another design method may have been a better choice, but we'll save that for another day!
